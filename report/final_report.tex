\subsection{Dataset split for training and validation}
A common technique to ensure that we can correctly evaluate our model's performance and tune hyperparameters accordingly is to train the model on a training set and, evaluate it on a validation set. To reduce biases caused by the particular choice of the training/validation split, we also used k-fold cross validation (with k=5), in which we repeat the training/validation process 5 times with different splits and average the results to get a more stable estimate of our model's performance. We chose k=5 because we used the typical 80/20 train/validation split for all our models (80\% of the data for training and 20\% for validation) to maximize the number of images we can use for training while still having a good amount of data for validation; with that split, 5 folds would allow us to use all of our data for training and validation across the different folds.

\section{Preprocessing of Data and Design Choices}
We choose to apply different preprocessing pipelines to our different model architectures, as they have different requirements and are affected differently by the preprocessing steps.
% For example, the CNN model is more robust to noise and can learn features from raw data, while the SVM model is more sensitive to noise and requires more careful feature engineering. Hence, we applied a more complex and aggressive preprocessing pipeline to our CNN model, while we applied a simpler and more targeted preprocessing pipeline to our SVM model. \par

\subsection{Preprocessing for the CNN Classifier} \label{sec:cnn_preprocessing}
All of the transforms we use in our CNN pipeline are located in the \texttt{torchvision.transforms} package. Our training pipeline consists of the following steps:
\begin{enumerate}
  \item \texttt{ToImage()}: convert the initial 224x224x3 image vector to a PyTorch image tensor, which allowed us to utilize CUDA and improve training/prediction speed.
  \item \texttt{Resize(224 * 1.1)}: upscale the image tensor to about 245 pixels to create some room for randomness and thus help prevent overfitting to exact framings of a picture. (Professor Baraniuk demonstrated similar methods in class.)
  \item \texttt{RandomCrop(224)}: randomly crop the image to its original size (224x224), to force our model to learn to identify an object ``anywhere'' in the image, not just in the center.
  \item \texttt{RandomHorizontalFlip(p=0.5)}: since butterflies/moths are often symmetrically shaped, sometimes horizontally flip the image to prevent the model from having a bias toward left/right facing butterflies.
  \item \texttt{RandomRotation(degrees=10)}: a butterfly/moth can also be arbitrarily oriented, so randomly rotate the image anywhere from -10\degree \ to 10\degree \  to prevent the model from overfitting to the vertical orientations of our butterflies.
  \item \texttt{RandomAffine(degrees=0, translate=(0.1, 0.1))}: since the butterfly/moth can also be arbitrarily translated in the image, randomly translate the image by up to 10\% of its size vertically and horizontally to prevent the model from overfitting to the centered position of our butterflies (for example, in the exposition pose images like figure \ref{fig:right}).
  \item \texttt{RandAugment(ops=2, magnitude=10)}: beyond simple effects like rotations, translations, and reflections, the RandAugment augmentation method utilizes other augmentations that we did not explicitly code our pipeline for. We felt this could especially be useful for further expand the variability of our dataset and thus encourage the model to learn truly useful features.
  \item \texttt{ToDtype(torch.float32, scale=True)}: convert the pixel values to floats between 0.0 and 1.0, which is a more stable input range for our model to learn from, especially with ReLU activations.
  \item\texttt{ColorJitter(brightness=0.1, contrast=0.1, saturation=0.1, hue=0.02)}: we wanted our model to tolerate and learn different shades of a colored butterfly/genetic variations/dirt etc., but not too much since butterflies are mainly distinguished by their color, so apply a small random variation to the hue, saturation, saturation, and contrast of the image.
  \item  \texttt{Normalize(mean=(0.5, 0.5, 0.5), std=(0.5, 0.5, 0.5))}: center and normalize pixel values to ensure that the input data works well with ReLU steps in our CNN and that the training produces more stable gradients.

    For our validation and test pipeline, we only convert the data to a Pytorch image tensor, ensure the image is 224x224x3, center and normalize, and convert to a float tensor. We do this because all the augmentation steps described above are useful in training the model, but we want consistent/stable validation, so we decided not to apply those preprocessing steps to our validation/test sets.
\end{enumerate}

\subsection{Preprocessing for Kernel SVM}
Our preprocessing pipeline for the kernel SVM model was quite different and perhaps more complicated than the one for the CNN model, partially because the CNN model is more robust to noise and can learn features from raw data, while the SVM model is more sensitive to noise and requires more careful feature engineering. Hence, we applied a simpler and more targeted preprocessing pipeline to our SVM model. In addition, we saw relatively poor initial results from the kernel SVM model compared to the CNN classifier, and we felt that while the SVM model could be potentially improved with more careful feature engineering, the CNN was already performing much better, so we decided to focus our effforts more on improving the CNN model rather than the SVM model. Our SVM preprocessing pipeline consists of the following steps:
\begin{enumerate}
  \item Resize the image to 64x64x3. We utilize scikit-learn for our SVM implementation, which meant that the SVM training/prediction process is sequential and CPU-heavy, so we wanted to reduce the number of features in our data vectors to improve training/prediction speed. We chose 64x64 because it is a reasonable balance between preserving the features of the image and reducing the number of features.
  \item We double the size of our dataset, generating a copy of the images which is horizontally flipped using \texttt{numpy.fliplr()}. This exploits the symmetry of butterflies and prevents our model from overfitting to left/right facing orientations.
  \item We then used the popular HOG feature descriptor (Histogram of Oriented Gradients) to extract edges, textures, and other shapes of the picture by computing gradients between pixels and comparing their directions. This was because the wing shapes play a huge role in differentiating butterflies, as we observed from our dataset.
  \item Unfortunately, HOG throws away color information, which can be hugely important in accurately classifying butterflies and moths. So we used color histograms using \texttt{numpy.histogram(bins=32)} to put the red, green, and blue pixel intensities into 32 bins and append 96 features indicating the colors in each image. This way, our model can use color data to classify butterflies more accurately.
  \item To avoid brightness/natural noise in colors, we normalized our color histogram values so that the color features are comparable between images.
  \item We then append the color histograms to the HOG features.
  \item We then normalize and center all of our data vectors, because SVMs (especially with the Gaussian kernel) can be extremely sensitive to disproportionate deviations in features.
  \item Finally, since HOG produces very large data vectors, we apply PCA to project all our data vectors onto the first $n$ eigenvectors that capture 95\% of the variance so that both training and validation speed improve.

    The validation/test pipeline was the exact same except that we did not flip the images, as this would just introduce randomness into the prediction process.

\subsection{Comparison of the preprocessing pipelines used}
\subsubsection{Feature extraction}
\begin{enumerate}
  \item \textbf{In our HOG+SVM model}, we deterministically computed edges using HOG and color distributions using histograms. This gave us a fixed representation of each image.
  \item \textbf{In our CNN model}, we let the neural network ``learn'' the features by detecting the edges in the early layers, the textures in the middle layers, the object parts in the deeper layers. This gave us a more adaptive representation of each image, likely superior to HOG+SVM; since if that fixed representation loses some information about the butterfly, the HOG and color features alone cannot recover it, while a CNN can learn more about the useful features - not relying on a fixed engineered feature set.
\end{enumerate}
\subsubsection{Data augmentation}
\begin{enumerate}
  \item \textbf{In our HOG+SVM model,} we only did a horizontal flip on the images. This is because edges do not change much by color/lighting changes and since HOG is a much lower capacity model (compared to neural networks), too much augmentation can drastically reduce model accuracy rather than improving it.
  \item \textbf{In our CNN model,} we applied a bunch of different random augmentations on our data (as mentioned above), ranging from cropping it to changing the contrasts. This is because even though CNNs learn the features of an image well, they might sometimes learn them \textit{too well}. We need a lot of variation in augmentation to prevent overfitting to the data, especially in a dataset like this where in many of the images the object (butterfly) is relatively in the center of the picture.
\end{enumerate}
\subsubsection{Local spatial information}
\begin{enumerate}
  \item \textbf{In our HOG+SVM model,} the preprocessing steps (HOG+color histograms) do not really tell us about the locations of a color in an image, which means we lose some spatial information while training. In non-neural-network pipelines, especially a HOG-based SVM, this is inevitable because HOG cannot differentiate colors, only shape, while separate preprocessing approaches like color histograms only differentiate color, not shape.
  \item \textbf{In our CNN model,} we use \textit{convolutions} to propagate the information regarding where particular colors appear in the image. Unlike the SVM approach, this helps us pinpoint how each part of a subject is colored and we can use that to differentiate it from other images.
\end{enumerate}
\subsubsection{Normalization}
\begin{enumerate}
  \item \textbf{In our HOG+SVM model,} we normalize the features of our data \textit{after} we are done with extracting the features. This is because HOG representations make the RBF kernel SVM extremely vulnerable to small disproportionate variations in features.
  \item \textbf{In our CNN model,} we normalize the pixels of our data \textit{before} we start learning the features. This is because the CNN needs a stable input range to learn the features and is much more stable to small variations compared to other approaches like the SVM.
\end{enumerate}
\subsubsection{Dimensionality reduction}
\begin{enumerate}
  \item \textbf{In our HOG+SVM model,} we apply PCA on our data vectors to capture 0.95 of the variance, because training and testing the SVM is a sequential, CPU-heavy process.
  \item \textbf{In our CNN model,} we do not externally apply any dimensionality reduction but let the layers of our neural network handle it. The pooling, convolutions, and compression inside the hidden layers do the ``dimensionality reduction'' for us. Moreover, we were able to parallelize the CNN training, such that it remains efficient with more features.
  \item Similarly, \textbf{in our KNN + SIFT}, we reduced the dimensionality of the data as SIFT returns only 196 parameters.
\end{enumerate}

\section{Models Considered}
\subsection{ResNet CNN}
\subsubsection{Why we chose to try it}
We have read some articles online that the maximum image classification accuracies achieved in diverse datasets like Caltech 101 were around 70--80\% without the use of convolutional neural networks (CNNs) \cite{svmacc}, while CNNs broke this barrier, attaining 90--99\% accuracy with ease \cite{imagenet, Zabir_Fazira_Ibrahim_Sabri_2018}. Also, the final project description explicitly states that one of our model choices shouldn't be a neural network, which suggested to us that a neural network might perform quite well. In addition, a CNN is a well-known architecture for approaching image classification problems like MNIST digit recognition or the ImageNet dataset, so we expected it to perform well on our butterfly classification problem as well.
\subsubsection{Algorithm/Workflow}\cite{resnet}
Like a plain neural network, the CNN passes images through a series of layers before ultimately predicting class probabilities with a softmax/multinomial logistic regression output layer. The advantage of the CNN is that it uses matrix convolution rather than explicit vector/array multiplication to significantly reduce the number of model parameters. We further decided to use \textbf{residual skip connections}, which allow input data to ``skip'' layers and be directly added to the activation output of a future layer. This approach mitigates the issue of gradients vanishing in backpropagation and allows us to significantly deepen the network, which is a desirable attribute.
\begin{enumerate}
  \item We pass preprocessed images to the network as a batch of tensors.
  \item We extract the low-level features using an initial convolutional layer, which is implemented with batch normalization and ReLU.
  \item We then pass the output to a stack of residual blocks, where each apply convolutions, normalization, ReLU and \textbf{skip connections}. It is mentioned in the lecture notes that skip connections help stability in optimization.
  \item With strided convolutions, the spatial resolution of our data shrinks, while the channel width increases - enabling us to represent more complex features of the data.
  \item After that, we perform average pooling to aggregate each feature map to a single scalar per channel, yielding a fixed length vector of size 512.
  \item Finally, the classifier head (with the help of softmax embedded in the cross entropy loss function) maps this vector to 100 logits, via a dropout followed by a linear layer to the class logits.
  \item Now that we are ready to score our vector, we optimize over the cross-entropy loss function we learned in class, with a good optimization algorithm (adamW, for example).
  \item We then backpropagate the gradients we get from our optimization algorithm, and update the weights of our model.
  \item We repeat this process until our loss converges, or until we hit the maximum number of epochs we set.
\end{enumerate}
\subsubsection{Design choices}
\begin{enumerate}
  \item \textbf{Amount of feature complexity through layers:} We want to gradually increase feature complexity through the layers of our neural network, so in the first layers, we want to learn simpler things like edges and color; but as we go deeper in the network, we also want to learn more specific things like the markings on a butterfly. So, we need to gradually increase feature sizes and complexity of the operations we perform in the layers and residual blocks. (For example, start with just linear layer + ReLU, and continue with multiple layers of ReLU + 1-norm normalization.)
  \item \textbf{How much to reduce image size as we process it:} We want easier to analyze, smaller representations of an image that preserves what is present in the image correctly. But we also do not want to lose the actual features of an image that differentiate it from the others. So, we need to choose the step by step scaling accordingly.
  \item \textbf{The number of residual blocks and layers we use:} We want a model that both accurately detects the subject and has high performance in training, validation and testing, also speed-wise. Therefore, we need to be careful about adjusting how many features each residual block introduces into the data, and how many of the layers/blocks we have. Too many layers and residual blocks can impact performance while introducing marginal accuracy/maybe causing overfitting, but too few can cause the model to underfit or become unstable. We experimented with different numbers of residual skip connections and different numbers of layers, eventually settling on a network very similar to the standard 18-layer ResNet.
  \item \textbf{Learning rate: scheduled or constant?} A learning rate too large can cause the model to become very unstable or overshoot the optimum, but one too slow can negatively impact the learning performance of our model. Hence, we should decide on what the initial learning rate is, and if we should decay the learning rate or not.
  \item \textbf{How to decay the model weights over time:} Too large model weights can cause the model to overfit, and too much decay in the weights can cause the model to underfit. We also need to implement a schedule on decaying the weights of our model to keep it consistent.
  \item \textbf{What to choose the batch size:} Since the optimization methods we use are usually based on batch optimization (like AdamW), we need to choose a batch size that is enough to produce a stable training process, but one that is not too large so that the training stays fast.
  \item \textbf{What to choose the loss function and optimizer:} Like in many other problems, the cross-entropy loss function is generally unmatched in its success here. So, we do not regard it as that flexible of a choice but still wanted to include it here. Coming to the optimization algorithm, we need an optimization algorithm that both incorporates fast early learning, and later fine-tuning. Moreover, it would help us if our optimizer decayed the weights/scheduled the learning for us. (Yes, we are hinting at AdamW, adaptive moment estimation with weight decay.)
  \item \textbf{What to choose the cross-validation parameter $k$:} The model accuracy and overfitting depends on the optimal choice of the cross-validation parameter, i.e. the number of partitions. If $k$ is too low, we can overfit heavily; but if it is too high, we gain little while losing training performance.
  \item \textbf{The preprocessing steps to the data input:} This is in many ways the most important part, as CNNs depend heavily on their input just like any other classification model. We have already discussed these choices and the steps we used in the previous section.
\end{enumerate}

\subsubsection{Course concepts used}
\begin{enumerate}
  \item \textbf{Supervised learning/Classification}: The ResNet CNN we trained is a supervised learning model where inputs are images (data) with labels and the outputs are a discrete number (100) of classes. Hence, this is a mainstream classification problem which we have learned in class to approach with methods like logistic regression, KNN, SVMs, and of course, neural networks.
  \item \textbf{Feature engineering and learning}: Our CNN learns the features of the data by applying convolution layers and residual blocks. So, in essence, it does the feature engineering for us.
  \item \textbf{Optimization}: We use a convex cross-entropy loss function, which we optimize with the AdamW optimizer, which uses gradient descent at its core.
  \item \textbf{Linear algebra background}: Since we are working with matrices, tensor operations, convolutional layers, max-pooling, and backpropagation of gradients, our work is based in ideas from vector calculus and linear algebra.
\end{enumerate}
\subsubsection{Loss function of the model}
Our CNN uses cross-entropy loss and uses the adamW optimizer in gradient descent. The cross-entropy loss is derived from Maximum Likelihood Estimation (MLE) of logistic regression, and it has two terms, where one activates and the other deactivates. It is also convex, and hence optimizable by gradient descent-based algorithms. adamW is a fairly new algorithm (2017, \cite{loshchilov2019decoupledweightdecayregularization}) that uses ``adaptive moment estimation'' and ``weight decaying'', in which the ``moments'' are closely related to gradients in gradient descent; and weight decaying is just a smart way of steering the model away from underfitting to more previous data in the training procedure. In addition to that, we use a cosine learning rate scheduler to help with its stability.
\subsection{Model 2: Kernel SVM}
\subsubsection{Why we chose to try it}
We chose the Kernel SVM as the required ``non-neural-network based model'' for the  project (in addition, we were required to implement one in Homework 7 of the course). We read some articles online that suggested with the right training regimen and preprocessing (HOG), kernel SVMs can achieve accuracies as high as 70\% in multi-class classification problems similar to our dataset (like Caltech 101)\cite{svmacc}, so it was worth trying to see how well it would perform on our butterfly classification problem.
\subsubsection{Algorithm/Workflow}
Support vector machines (SVMs)  learn a hyperplane to separate the data points. Moreover, to increase model stability, they try to maximize the margin of the hyperplane by learning a ``hyper-slab'' and maximizing the width of that slab. This ensures that the classification model is more stable for the future data to come, even if it trades off some accuracy for the training data while doing so. Our data is almost always not linearly separable in the original low number of dimensions, so we use the Gaussian kernel to implicitly transform the data into infinite dimensions (which is generally linearly separable given a finite amount of data, guaranteed by the Stone-Weierstrass theorem discussed in lecture) while using the kernel trick to optimize our loss function and train the SVM efficiently. The workflow is as follows:
\begin{enumerate}
  \item Preprocess the data as described in the previous section.
  \item We use the Gaussian kernel to map our data to ``infinite'' dimensions.
  \item Using the kernel trick, minimize our loss function, which adds the hinge loss and the margin maximization, to learn the parameters of our SVM model. That loss function looks like the following:
    \[\mathcal{L}(\mathbf{\alpha}, b) = \sum_{i=1}^{n}(1-y_i(\mathbf{\alpha^\intercal X x}_i + b))_+ +  \lambda \norm{\mathbf{X^\intercal \alpha}} \] where $\alpha$ and $b$ are the parameters of our SVM model, $y_i$ is the label of the $i$-th data point, $\mathbf{X}$ is the matrix of our data points, and $\lambda$ is the regularization parameter that controls how much we want to maximize the margin vs. minimize the hinge loss.
  \item While training, depending on $\lambda = 1/C$, we allow some of the points be misclassified or lie on the margin, for which we call them our \textbf{support vectors}.
  \item We then use the support vectors to implement our decision function, in which the decision function looks like as follows ($SV$ is index set of training points with nonzero slack variable, i.e. exactly the support vectors):
    \begin{center}
      \(f(\mathbf{\hat x}) = \sum_{i \in SV} \alpha_i y_i K(\mathbf{\hat x}_i, \mathbf{x}) + b\)
    \end{center}
  \item We predict the class of the validation/test set. i.e.:
    \begin{center}
      \(\hat{y} = \operatorname{argmax}_{k} \; f_k(\mathbf{\hat x})\)
    \end{center} where $k$ is the class index.
\end{enumerate}
\subsubsection{Design choices}
\begin{enumerate}
  \item \textbf{Kernel choice:} We need a kernel that has enough dimensionality to get us a linearly separable transformation, but also one that prevents overfitting and is stable. The gaussian kernel is known to have both of those features. By contrast, the sigmoid (tanh) kernel is regarded to be unstable (since the kernel matrix is not guaranteed to be positive semi-definite) even though it also maps the data to ``infinite'' dimensions; and the polynomial kernels are known to overfit/underfit easily to complex datasets.
  \item \textbf{Regularization parameter $C$:} Do we want higher bias, lower variance (higher $C$) in our hyper-slab, or vice versa (lower $C$)? This can drastically change our prediction accuracy, and we adjusted this hyperparameter to get higher prediction accuracy/lower overfitting levels.
  \item \textbf{Gaussian exponentiation scaler $\gamma$:} If we increase this value, even the slightest spatial local adjustments between data can affect our classifier, which can lead to overfitting. However, if we decrease this value, our classifier may underfit. We also need to pay attention to this hyperparameter to have a good classifier.
  \item As with the CNN model, the preprocessing of the data is the most important of the design choices we have made. Most of the work we did in training this classifier was considering how best to preprocess the data (finally deciding to HOG+PCA). If the input data does not provide a good representation of the images we want to classify, the SVM will perform poorly no matter how well we tune the hyperparameters of the SVM.
\end{enumerate}
\subsubsection{Course concepts used}
\begin{enumerate}
  \item \textbf{Supervised learning/Classification}: Similarly to the CNN model, we do supervised learning here, and the problem is a classification problem.
  \item \textbf{Optimization/SGD}: We solve a convex optimization problem by minimizing hinge-loss while maximizing margin in the SVM. We are using a variant of (stochastic) gradient descent to solve this problem.
  \item \textbf{Unsupervised learning/PCA}: This is only used in the preprocessing step to make our training process smoother and faster.
  \item \textbf{Feature engineering}: While this is also only used in the preprocessing step, our decision to utilize HOG and color histograms is crucial to our model's performance and we need a good manual design of features to accurately represent our data when using an SVM.
  \item \textbf{Kernel methods/Kernel trick}: To make our data vectors linearly separable, we nonlinearly transform the data using the RBF kernel, and practically optimize the SVM using the kernel trick.
  \item \textbf{Linear algebra background:} We are also working with matrices and vector operations here both in the preprocessing of the data and the training of the model. We also use gradients in the optimization for this model, so our work is still based in vector calculus and linear algebra.
\end{enumerate}
\subsection{Model 3: SIFT + KNN}
\subsubsection{Why we chose to try it}
When we started looking for filters for our image processing step, we noticed that the SIFT feature descriptor (introduced in class) reduced each image from several thousand features (\(224^2 \cdot 3 = 150528\) pixels per image; even if resized: \(64^2*3 = 12288\) pixels per image) down to 196. On the other hand, the K-Nearest Neighbors classification algorithm is known for accuracy but also for its high time complexity, so we hypothesized that we could reduce the dataset's feature size using SIFT and then classify with KNN.

Furthermore, as the points of interest are scale and rotation invariant, this improves KNN ability to deal with rotated pictures (which was a severe issue as it changes almost all the pixels around).
\subsubsection{Issues we ran into}
Ultimately, after limited testing (as aforementioned, we invested the bulk of our time into improving our CNN model as it showed good results early on), this model achieved at best 42\% validation accuracy. \par
We considered other ways to improve this model. However, we ran into a few issues, the most important one being that SIFT does not account for any color information, an attribute likely essential for classifying butterflies. Moreover, when we started looking into other possible filters, almost all of them increased the image complexity---which made KNN too inefficient for classification. Hence, we had extremely limited options in terms of possible preprocessing pipelines, and hyperparameter tuning (the only one being the number of neighbors). Lastly, optimizing this algorithm is also harder, as there is no loss function to apply gradient descent on.\par

Thus, given that the performance (at best around 40\%) was considerably worse than the our simple initial implementation of SVM, we decided not to proceed further with this architecture. \par

\section{Training Procedures}
We describe the training and validation process of our models as implemented in our Python code in this section (PyTorch Lightning notebook \texttt{kfold\_clf.ipynb}, \texttt{svmkernel\_model.py}, and \texttt{sift\_knn\_classifier.py}). Preprocessing and data augmentation are already summarized in the previous sections; hence, here we focus on data splits, optimization, schedules, regularization, and how we trained each classifier.

\subsection{Model 1: ResNet CNN (with PyTorch Lightning)}
We used PyTorch Lightning to train our CNN classifier, as the library allowed us to streamline our code, as well as quickly iterate training the model with different hyperparameters and maintain comprehensive logging of the training process. The model architecture is based on ResNet-18, which has 18 layers with three structural changes compared to the ImageNet: the first convolution uses stride of 1 and padding of 1 (we downsample later compared to the usual ResNet architectures); the max-pooling stage is replaced by an identity mapping to preserve more fine-grained information; and the original fully connected layer is replaced by a two-part head dropout block with probability 0.4 followed by a linear mapping from 512 features to 100 classes (the output number of classes). The model applies dropout only at the head and not inside the residual blocks.
\begin{enumerate}
  \item\textbf{Data partitioning.} We use scikit-learn's \texttt{StratifiedKFold} with shuffling (seed 42 for reproducibility), and \texttt{n\_splits}=6 for 6-fold cross validation (even though we mentioned 5-fold above, averaging across our 6-fold results proved marginally superior in our experiments). This way, we train and test our classifier by averaging out the parameters learned from the whole dataset, yielding a more stable training and test regimen.
  \item\textbf{Mini-batch training.} For each fold in our training and validation process, \texttt{ImageDataset} maps CSV rows to integer labels via a fixed alphabetical ordering of class names. The training dataloader uses a batch size of 32, random shuffling, \texttt{num\_workers}=4 for faster training/prediction, and \texttt{pin\_memory=True} for increased performance. Validation and test prediction loaders use the non-augmented preprocessing pipeline aforementioned since they should make predictions on the raw data; and the test \texttt{DataLoader} uses a batch size of 1 to make predictions sequentially.
  \item\textbf{Class imbalance in the loss.} Before we fit a fold, we compute class frequencies on the fold's training subset. We set the class-weights proportional to the inverse frequency (total count divided by class count, scaled by the number of classes) and pass them into \texttt{torch.nn.CrossEntropyLoss} as tensor weights. This reweights the softmax cross-entropy so we counter class imbalance in the loss by making species with less number of samples contribute relatively more to the gradients.
  \item\textbf{Loss, label smoothing, and optimization.} The training configuration sets label smoothing to 0.1 combined with the weighted cross-entropy approach we discussed above. This replaces one-hot truth label vectors by smoother versions, ideally improving the model's generalizability and accuracy. Our optimizer is adamW with initial learning rate $3\times10^{-3}$ and weight decay $10^{-2}$, since Adam is widely regarded as one of the best optimization algorithms with weight decay helping stability in convergence.\cite{loshchilov2019decoupledweightdecayregularization} A cosine annealing scheduler (\texttt{CosineAnnealingLR}) runs through the maximum number of epochs with \texttt{T\_max = 100}, stepping once per epoch - this was the best scheduler among the three we tried (alongside \texttt{CosineAnnealingWarmRestarts} and \texttt{ReduceLROnPlateau}). We also enable mixed-precision training when supported (\texttt{16-mixed}) for increased performance and better memory complexity; and we also set \texttt{torch.set\_float32\_matmul\_precision('medium')} for the Tensor Core behavior on the utilized NVIDIA GPUs to help speed up the training process.
  \item\textbf{Stopping, checkpoints, and logging.} We use early stopping in our training process based on the validation accuracy (we could also have used the validation loss) with a patience  of 12 epochs (i.e. wait 12 epochs of no improvement) to prevent overfitting. We continually store the top three checkpoints by validation accuracy, as well as saveing the last checkpoint so we can pause and resume training as needed. We also comprehensively log the training and validation loss/accuracy and optimizer learning rate/momentum directory for each fold of the training process and use a learning rate monitor and a progress bar for easier analysis of checkpoint convergence.
  \item\textbf{Ensembling across folds.} After training all folds, we load the checkpoints and either average the logits across folds or take a per-sample majority vote among the argmax predictions from each fold (averaging proved more successful in practice). After that, we map the indices back to species names for submission. We use this approach to exploit the diversity of our dataset and have a more stable classifier since it has been empirically shown in studies that ensembling can help increase accuracy while also improving overfitting \cite{8539396}.
\end{enumerate}
\subsection{Model 2: HOG + RBF kernel SVM (\texttt{svmkernel\_model.py})}
Unlike our ResNet CNN model, we do not use $k$-fold cross validation in this model - because the initial results of our CNN model yielded much higher performance and accuracy and we read the literature on the limitations of non-neural-network approaches (especially about having significantly lower accuracy compared to neural networks in \textbf{image recognition})\cite{WANG202161}. However, if we wanted to further improve this model, we would definitely implement it. We still map the labels to integers using a scikit-learn \texttt{LabelEncoder}.

We use scikit-learn's implementation of the SVM classifier with the Gaussian kernel (seed 42), and set the hyperparameters to the implementation's defaults, $C=1$ and $\gamma$ to the ``scale'' heuristic which adapts the kernel width to the feature scale of the training data. We fit the model on our full PCA-reduced, augmented training set in one batch solve.

\subsection{Model 3: KNN + SIFT (\texttt{sift\_knn\_classifier.py})}
Our pipeline for this model is light and uses mostly already implemented packages, as this is more of an experiment because Prof. Baraniuk stated in class that K-NN performs well in general even though the complexity is high. We use OpenCV's SIFT implementation after grayscaling the images in the training dataset. We then standardize the variable length SIFT output to vectors of fixed length and concatenate the mean and standard deviation of all 128 rows to the data, giving 256 features per image. If no keypoints or descriptors are returned, we substitute that with a zero vector - but this rarely happens in our dataset from our observations after training the model.

To reduce runtime, we only ``sample'' every fifth row of \texttt{train.csv}. We skip (\texttt{idx \% sample\_rate != 0} with \texttt{sample\_rate}=5), so, roughly 20\% of labeled data are used as train data when making predictions. This is to increase performance since as we mentioned, KNN is a slow classifier with high time complexity. We standardize the features with scikit-learn's \texttt{StandardScaler} on that subset. We use scikit-learn's \texttt{KNeighborsClassifier} with the default $K=5$ neighbors and Euclidean distance metric. We do not perform an explicitly training/validation split on this dataset to keep complexity of the model low, and performance high.

\section{Overfitting and Generalization Analysis}
\subsection{Overfitting in ResNet CNN}
\subsubsection{Evidence of overfitting}
The main evidence of overfitting in our model, was that we had a training accuracy of over 99\%, and a validation accuracy of 93\% (our final submission accuracy was somewhat higher, but we achieved that by using an ensemble averaging model across our different data splits). So, this is the main trait of overfitting, improved performance on training set, which did not improve the actual model performance.
Likewise, when it came to the SVM, we had a higher training accuracy than the validation one. In order to prevent this, we added a hyperparameter to punish larger  weight values ($\lambda$).

\subsubsection{What may have caused it}
A convolutional neural network has a relatively high capacity to represent features with respect to the dataset size per class. So, it may memorize the fine-grained visual details of the training images too much - like a child only seeing colored rabbits and thinking it is a different animal after being shown an albino one. In addition to that, many classes share similar wing shapes and only differ by subtle color differences, which increases sensitivity to small variations in images.

\subsubsection{How we tried to overcome it}
We reduced the overfitting of our neural network by tuning the following aspects - all of the reasons are already explained in above sections:\begin{enumerate}
    \item We performed data augmentation such as random crop, flip, rotation, affine, RandAugment, ColorJitter.
    \item We performed dropout with 0.4 probability in the classifier head.
    \item We used the weight decay version of adamW.
    \item We smoothed the labels by 0.1.
    \item We weighted the cross-entropy gradients with respect to classes.
    \item We performed early stopping on validation accuracy (12 epochs threshold).
    \item We performed ensembling on our folds at inference.
\end{enumerate}
\subsubsection{`Post-processing' overfit}
It is also worth noting that the neighbor bias we applied probably leads to significant overfitting, as the dataset we used to validate (the Kaggle leaderboard one) contains a very peculiar distribution (which although has some expected traits, was rather extreme in the sense that it was entirely sequential in its label).

\subsection{HOG + RBF Kernel SVM}
\subsubsection{Evidence of overfitting}
There was also significant overfitting in our SVM. When we fit the SVM, the training error we obtained was close to 96\%, but once we tried validating it, the results were almost 40\% worse - with an accuracy around 56.1\%. 
\subsubsection{What may have caused it}
The SVM heavily relies on our understanding of data through the features we engineer, which are the HOG and color histograms. If the engineered features miss the separating differences between the images or encode common `features' of the dataset, the performance can be significantly worse in the validation split. Also, the omission of $k$-fold cross-validation is another plausible explanation for the overfitting we experience here, showing the performance-accuracy tradeoff mentioned in class.
\subsubsection{How we tried to overcome it}
We mainly used feature scaling and horizontal flip augmentation on training data, and `regularization' for our RBF SVM (default parameters $C$ and $\gamma$ with scale schedule). Although these steps overcame some stability issues, they largely did not help reduce overfitting as significantly as in the CNN case.


\subsection{KNN + SIFT}
\subsection{Evidence of possible underfitting}
Our KNN + SIFT classifier had substantially lower performance on all sets (training 48.5\%, validation - i.e. all images - 29.8\%, Kaggle 37\%), which indicates our classifier may not have captured enough features to properly separate classes for our dataset.
\subsection{What may have caused it}\begin{enumerate}
    \item The SIFT feature extraction technique completely discards any color information, which is \textbf{one of the critical aspects describing butterfly species, as described in the `Data Exploration' section}.
    \item We downsampled the training subset to train on every 5th image to increase performance, which reduced available supervision considering species do not have many representative samples in the dataset.
    \item The KNN classifier has limited ability to learn nonlinear class boundaries in such settings where there are many classes with low number of distinguishing factors (since colors were omitted).
\end{enumerate}
\subsection{How we tried to overcome it}
We tested background removal and standardizing the features, but the effect of these approaches were limited on the overall accuracy. Since the performance remained significantly lower compared to the CNN and SVM, we did not proceed with more fine tuning of this model.

\section{Pipeline for the Final Submission}
Our best and final submission is our customized 100-class ResNet-18 CNN architecture. Here we describe the exact pipeline involved in the final submission. 

For each fold, we following the described steps. First, we package the dataset in a custom Pytorch Lightning \texttt{LightningDataModule} which contains all the necessary attributes of our dataset. This internally applies the preprocessed steps described in Section \ref{sec:cnn_preprocessing} to the training, validation, and prediction (for submission generation) dataloaders. Using a batch size of 32, 4 workers for data loading and training, a maximum of 100 epochs, initial learning 0.003, weight decay 0.01, label smoothing 0.01, and an early stopping patience of 12 epochs, we begin the training process (near the end of the project, we began using 3 Kaggle GPUs in parallel to speed up training; earlier, we only used a single NVIDIA 4050 GPU) while logging loss and accuracy metrics to Tensorboard and a CSV file. We used Pytorch Lightning's \texttt{Trainer} to easily configure this setup.

After training is complete on each fold, we load each fold's best checkpoint and for each test image we retrieve only the unnormalized 100-class probability vector (the ``logits'') and average this result across all folds. We increase the probabilities according to the neighbor bias as a post-processing step, as explained in Section \ref{sec: post-proc}. We take the argmax of that final probability vector and map the predicted index back to a pregenerated dictionary between indices and categorical species label contained in the \texttt{LightningDataModel}. We do this for each data point and save the prediction output to a submission file.

\section{``Post-Processing''} \label{sec: post-proc}
(Note for grader: we asked on Piazza whether this is a valid technique, and the instructor response said it was and would be valuable to discuss at length in the report.)

After we looked at the Kaggle test images, we noticed an odd pattern: the species were presented in sequential order (e.g. all the ADONIS class butterflies are presented consecutively). This makes sense, considering that if we were to look at a user feeding data into some form of database, they would probably be presented in clusters, since someone might take multiple pictures of the same butterfly. Furthermore, it is expected that the classification would have some natural statistical biases; for example, we might expect that if a past butterfly is from a certain location, then other species in the same region would be more likely. Thus, it makes sense to also account for those patterns when creating a classifier.

Hence, we altered our prediction pipeline for the CNN only (due to time considerations). Once we had our raw predicted probability vector for a data point, we retroactively scaled the softmax probability vector by a (1- 2$P$) and added P (a tunable hyperparameter) to the probability of the neighboring butterflies. With that, we managed to increase our final accuracy from 97.2\% to 99.8 \% accuracy on the Kaggle dataset. \par
It is worth noting though that this also increases the chance of overfitting to the particular dataset that we were working with, If someone decided to shuffle the data, the accuracy would then go down quite a lot; so, it is important to tune the P value. \par
